01

執行摘要

v3 在 v2 的 Serverless 基礎上，加入完整的 Modern Data Stack 分析層。 從原本「資料採集 → AI 摘要 → LINE 推播」的線性流程， 升級為「Hot Path（即時推播）+ Warm Path（SQL 分析 + 視覺化）」的**雙軌資料生命週期**。

新增的分析層採用 Medallion Architecture（Bronze / Silver / Gold 三層）， 透過 dbt 在 Amazon Athena 上做 SQL 轉換， 產出的 Gold Layer 直接供 Power BI / Tableau 連接建構互動式儀表板。

架構轉變後，本專案完整展示了 NoSQL + SQL 雙能力、 Lambda Architecture（Hot/Warm Path）、 Data Lakehouse、dbt 轉換、BI 整合等 企業級 Sr DE / Cloud Architect 全棧能力。

🆕 v3 相對於 v2 的關鍵升級

  • 新增 Stage 6 分析層：Athena + Glue Catalog 提供 Serverless SQL 查詢能力
  • 導入 Medallion Architecture：明確 Bronze（raw）→ Silver（cleaned）→ Gold（marts）三層
  • 導入 dbt 轉換層：跨股票聚合、回測勝率、業務邏輯都用 SQL 寫，享有版本控制與測試
  • 新增 dbt 資料品質測試：not_null、relationships、自訂規則自動跑 CI/CD
  • Power BI / Tableau 整合：透過 ODBC 連接 Athena，4 個核心 dashboard 規劃
  • 確認不加 RDBMS：BI 需求由 Athena 滿足，避免 RDS 維運成本與架構偏離
  • 專案資料夾新增 dbt/：完整 dbt project 結構
  • Roadmap 重排：dbt + BI 從 Phase 2 提前到 Phase 1 必做

三層儲存策略（Hot / Warm / Cold）

資料生命週期管理：不同存取模式對應不同服務，控制成本同時最大化分析能力

層級 服務 查詢延遲 適合場景 本專案用途
🔥 Hot DynamoDB < 10ms 即時點查、KV 存取 LINE Bot 即時查詢、訂閱管理、最新分析
🌤️ Warm S3 Parquet + Athena 1-10 秒 SQL 分析、跨表聚合 歷史回測、dbt 轉換、Power BI 接入
🧊 Cold S3 Glacier Deep Archive 數分鐘至 12 小時 歸檔保存 90 天以上原始資料、法遵保存

🎯 重要設計決策：為什麼不加 RDS？

  • BI 需求由 Athena 滿足：Power BI 透過 ODBC 直連 Athena，不需要 RDS
  • 成本考量：Aurora Serverless v2 最低 ~$45/月 24 小時掛機；Athena 按掃描量 ~$0.15/月
  • 避免架構偏離：純 Serverless 設計，不引入需要 instance management 的元件
  • 未來保留彈性：若 Phase 2 整合 OpenMetadata 時再考慮加 Aurora 當其後端

數據採集與 ETL（Bronze + Silver 雙寫）

🎯 Worker Lambda 同時寫入 Bronze（raw）與 Silver（cleaned），為 dbt 提供乾淨資料源
資料源 Strategy Pattern

TWSE OpenAPI 主源 → FinMind 備源 → yfinance 最終備援，自動 fallback。

Bronze 寫入：原汁原味存底 v3 強化

原始 API JSON response 寫入 s3://twstock-raw-prod/source=twse/year=2026/month=05/day=24/，不做任何處理。用途：法遵、ETL 重跑、資料追溯。

Silver 寫入：清洗 + 技術指標 v3 強化

清洗工作：型別轉換（string → float）、缺值處理、欄位命名標準化、計算 MA5/20/60、RSI、KD、MACD。

輸出至 s3://twstock-curated-prod/stock_features/year=2026/month=05/day=24/features.parquet，Snappy 壓縮。

DynamoDB 同步寫入

同樣資料寫入 DDB 給 Hot Path 使用，搭配 attribute_not_exists 條件式寫入確保冪等性。

AI 智能分析 (LLM Integration)

🎯 Bedrock Claude 4.5 Haiku 把技術指標翻譯成人類可讀摘要，含 Output Validator
模型：Claude 4.5 Haiku

低成本（$1/1M input tokens）、推理品質夠用、Bedrock ap-northeast-1 已上架。

Prompt + Grounding

System Prompt 設定為「資深量化分析師」，強制只使用輸入 JSON 中的數據，禁止推測未提供資訊。

Output Validator

Pydantic 驗證 Bedrock 回傳 JSON schema；confidence < 0.6 時走 fallback 規則引擎。

分析層與資料治理

🎯 將 Silver 層資料透過 dbt 加工為 Gold marts，供 BI 工具與回測使用

6.1 Medallion Architecture（Bronze / Silver / Gold）

採用 Databricks 提出、業界標準的三層 Data Lakehouse 設計：

🥉 Bronze Layer — Raw Data
內容原始 API response，無任何處理
格式JSON
位置s3://twstock-raw-prod/
用途法遵、ETL 重跑、資料追溯
寫入者Worker Lambda（Stage 2）
↓ Worker Lambda 清洗、型別正規化、技術指標
🥈 Silver Layer — Cleaned / Conformed
內容型別正確、欄位統一、缺值處理、含技術指標
格式Parquet + Snappy 壓縮
位置s3://twstock-curated-prod/stock_features/
用途dbt 源資料、ML 訓練資料、Athena 大部分查詢
寫入者Worker Lambda（Stage 2）
↓ dbt 跨表 JOIN、聚合、業務邏輯
🥇 Gold Layer — Business Marts
內容為「特定分析問題」量身打造的表，含聚合與業務指標
格式Parquet（dbt 產出）
位置s3://twstock-marts-prod/
用途BI 工具直接讀、Dashboard、報表
寫入者dbt run（每日 16:00 觸發）

6.2 Athena 配置

Athena 是 Serverless SQL Query Engine（底層 Presto），對 S3 上的資料執行 ANSI SQL，不儲存資料。

Glue Catalog 設定

DatabaseTables用途
twstock_silver stock_features, foreign_trades 對接 S3 Silver Parquet
twstock_gold fct_daily_signals, fct_backtest_returns, dim_stocks, mart_dashboard 對接 S3 Gold marts（dbt 產出）

Athena 查詢範例（直接寫 SQL！）

SQL on Athena
-- 回測：過去一年 RSI < 30 進場、5 日後的勝率 WITH signals AS ( SELECT stock_id, trade_date, close AS entry_price, LEAD(close, 5) OVER (PARTITION BY stock_id ORDER BY trade_date) AS exit_5d FROM twstock_silver.stock_features WHERE rsi_14 < 30 AND trade_date >= 20250524 ) SELECT COUNT(*) AS signal_count, SUM(CASE WHEN exit_5d > entry_price THEN 1 ELSE 0 END) AS wins, AVG((exit_5d - entry_price) / entry_price * 100) AS avg_return_pct FROM signals WHERE exit_5d IS NOT NULL;

6.3 dbt 專案設計

dbt（data build tool）是 SQL-based 的 transformation 框架，支援版本控制、測試、文件、依賴管理。

專案結構

tw_stock_bot/dbt/ ├── models/ │ ├── staging/ # 從 Silver 對接的入口 │ │ ├── stg_stock_prices.sql │ │ ├── stg_foreign_trades.sql │ │ └── _stg_schema.yml # 欄位定義 + 測試 │ ├── intermediate/ # 中介計算表 │ │ ├── int_daily_signals.sql │ │ └── int_signal_returns.sql │ └── marts/ # 🥇 Gold Layer 最終產出 │ ├── fct_daily_signals.sql # 今日所有訊號彙總 │ ├── fct_backtest_returns.sql # 訊號歷史勝率 │ ├── dim_stocks.sql # 股票主檔含產業 │ └── _marts_schema.yml ├── tests/ # 自訂資料品質測試 ├── macros/ # 可重用 SQL 函式 ├── dbt_project.yml └── profiles.yml # Athena 連線設定

dbt model 範例（Gold Layer）

models/marts/fct_backtest_returns.sql
{{ config( materialized='table', partitioned_by=['signal_type'], external_location='s3://twstock-marts-prod/fct_backtest_returns/' ) }} WITH signal_events AS ( SELECT stock_id, trade_date, signal_type, close_price AS entry_price, LEAD(close_price, 5) OVER ( PARTITION BY stock_id ORDER BY trade_date ) AS exit_price_5d FROM {{ ref('int_daily_signals') }} WHERE signal_type IS NOT NULL ) SELECT signal_type, COUNT(*) AS total_events, SUM(CASE WHEN exit_price_5d > entry_price THEN 1 ELSE 0 END) AS wins, ROUND(AVG((exit_price_5d - entry_price) / entry_price * 100), 2) AS avg_return_pct FROM signal_events WHERE exit_price_5d IS NOT NULL GROUP BY signal_type

dbt 資料品質測試

models/staging/_stg_schema.yml
version: 2 models: - name: stg_stock_prices columns: - name: stock_id tests: - not_null # ❗ 不允許 NULL - relationships: # ❗ 必須存在於 dim_stocks to: ref('dim_stocks') field: stock_id - name: close_price tests: - not_null - dbt_utils.expression_is_true: expression: "> 0" # ❗ 收盤價必須 > 0 - name: rsi_14 tests: - dbt_utils.expression_is_true: expression: ">= 0 AND <= 100" # ❗ RSI 必須 0-100 - name: trade_date tests: - not_null - dbt_utils.expression_is_true: expression: "<= CURRENT_DATE" # ❗ 不能是未來

每次 dbt test 都會跑這些檢查，資料異常時 CI/CD pipeline 直接 fail，不會把髒資料推給下游。

dbt 三層職責分工
